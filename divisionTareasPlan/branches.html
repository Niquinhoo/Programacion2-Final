Arquitectura de ramas GitHub — Restaurante Java

5 integrantes · Estrategia GitFlow adaptada · Capas heterogéneas e integración controlada

Rama Responsable Cuándo existe Propósito Merge hacia
main Todo el equipo Siempre Solo código funcional y entregable. Nadie pushea directo acá. —
develop Todo el equipo Siempre Integración continua. Todas las capas confluyen acá antes de ir a main. main
layer/model Integrante 1 Semana 1 POJOs, ConexionDB Singleton, schema.sql, seed.sql, Validaciones. develop
layer/dao Integrante 2 Semana 2 Todos los DAO con JDBC. Depende de layer/model mergeado en develop. develop
layer/service-controller Integrante 3 Semana 3+ Services con lógica de negocio y Controllers con ActionListeners. develop
layer/view Integrante 4 Semana 3+ Vistas Swing del cliente/mozo: menú, pedido, mesas, login. develop
layer/admin-reports Integrante 5 Semana 3+ ABM de productos, reportes, exportación PDF, pom.xml y JAR. develop
hotfix/nombre Quien lo detecta Si surge un bug Corrección urgente. Sale de main, vuelve a main y develop. main + develop
1
Crear el repo y las ramas base
Un integrante crea el repo en GitHub, hace el commit inicial con el pom.xml y el .gitignore, y crea develop desde main. Todos clonan y crean su rama de capa: git checkout -b layer/model develop.
2
Integrante 1 termina layer/model y hace Pull Request a develop
Cuando los POJOs, ConexionDB y el schema.sql están listos, abre un PR de layer/model → develop. Al menos un integrante revisa y aprueba. Después del merge, todos hacen git pull origin develop para sincronizarse.
3
Integrante 2 termina layer/dao y hace PR a develop
Mismo flujo. Antes de arrancar, el Integrante 2 hace git rebase develop en su rama para tener los modelos frescos. Al mergear, los integrantes 3, 4 y 5 también hacen pull de develop para tener los DAOs disponibles.
4
Integrantes 3, 4 y 5 trabajan en paralelo
Cada uno trabaja en su rama independiente. Si necesitan actualizar con cambios de develop: git rebase origin/develop. Cuando terminan un bloque funcional (no necesariamente todo), abren un PR parcial a develop. Así se integra de a poco y se detectan conflictos temprano.
5
Release: merge develop → main con tag de versión
Cuando la app funciona end-to-end en develop, se hace el merge final a main y se crea el tag: git tag v1.0 -m "Entrega final". El JAR ejecutable se genera con mvn package y se sube como Release en GitHub.

Reglas que hacen funcionar la arquitectura

🚫Nadie pushea directo a main ni a develop. Todo entra por Pull Request.
✅Cada PR necesita al menos 1 aprobación de otro integrante antes de mergearse.
🔄Sincronizar con develop frecuentemente: git rebase origin/develop en la rama propia evita conflictos grandes al final.
📝Commits descriptivos en español o inglés, consistentes: feat: agregar PedidoDAO con insert y listar, fix: calcularTotal con productos eliminados.
⚠️Si hay un bug crítico en main, se crea hotfix/nombre desde main, se corrige, y se mergea a main y a develop para que nadie trabaje sobre el bug.
🏷️Al final de cada semana, etiquetar el estado de develop: git tag semana-1. Sirve para mostrar progreso en la presentación.
