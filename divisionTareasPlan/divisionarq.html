Cómo se integran los repos sin red

1
Backend define su groupId y versión en pom.xml
En restaurante-backend/pom.xml se pone <groupId>com.restaurante</groupId>, <artifactId>restaurante-backend</artifactId>, <version>1.0</version>. Cuando alguien hace mvn install, Maven deposita el .jar en el repositorio local ~/.m2 de cada máquina.
2
Frontend lo importa como dependencia Maven
En restaurante-frontend/pom.xml se agrega la dependencia al backend. Maven la resuelve desde el ~/.m2 local — no hace falta red ni servidor. Cada integrante del front solo necesita haber corrido mvn install en el repo de backend una vez.

<dependency>
  <groupId>com.restaurante</groupId>
  <artifactId>restaurante-backend</artifactId>
  <version>1.0</version>
</dependency>
3
restaurante-app empaqueta todo con maven-shade-plugin
El tercer repo tiene un pom.xml que declara como dependencia al frontend (que ya carga al backend transitivamente). El maven-shade-plugin genera un único uber-jar con todo adentro — clases de backend, frontend, MySQL connector y recursos. Con mvn package se produce el restaurante-v1.0.jar ejecutable que se sube como GitHub Release.

# Sistema de Restaurante — Java

Proyecto universitario · Arquitectura tri-capa · Swing + Maven + MySQL

## Repositorios

| Capa | Repositorio | Responsables |
|------|-------------|--------------|
| Backend (model, dao, service, controller) | [restaurante-backend](https://github.com/restaurante-org/restaurante-backend) | Integrantes 1, 2 y 3 |
| Frontend (vistas Swing) | [restaurante-frontend](https://github.com/restaurante-org/restaurante-frontend) | Integrantes 4 y 5 |
| Presentación + entregable | [restaurante-app](https://github.com/restaurante-org/restaurante-app) | Todo el equipo |

## Descarga y ejecución

```bash
java -jar releases/restaurante-v1.0.jar
```

## Requisitos
- Java 17+
- MySQL 8 corriendo localmente
- Ejecutar schema.sql y seed.sql antes del primer arranque

Reglas clave para que los 3 repos convivan

🔒Nadie pushea directo a main en ningún repo. Todo entra por PR con al menos 1 aprobación.
📦Cada vez que backend mergea algo a su main, los integrantes del frontend corren mvn install en el repo de backend para actualizar su ~/.m2.
🏷️Backend versiona con tags: v1.0, v1.1. El frontend actualiza el número de versión en su pom.xml de forma explícita y consciente, no automática.
🤝Integrante 3 tiene acceso a ambos repos porque trabaja en layer/service-controller del backend pero coordina con el frontend. Es el punto de contacto entre ambos equipos.
🚀El repo restaurante-app solo se toca para releases. Nunca tiene ramas de feature ni de capa — solo main.
Java puro · sin red entre procesos · Maven local como puente entre repos · GitHub Releases como entregable
